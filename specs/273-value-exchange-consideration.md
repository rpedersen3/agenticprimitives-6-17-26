# Spec 273 — Value Exchange Kernel: ResourceItems, Orders, Legs (money is one resource type)

**Status:** Drafted (2026-06-11); expanded same day after external design review (Seaport / AP2 / ERC-7683 patterns).
**Owns:** The exchange kernel — `ResourceItem`, `ExchangeOrder` (Layer 5), consideration legs (Layer 8), zone validators, resource adapters, and the outcome-receipt evidence chain. The typed structure that makes the substrate **exchange-centered** (barter-capable) rather than payment-centered.
**Architecture-of-record:** [ADR-0024](../docs/architecture/decisions/0024-intent-coordination-substrate.md) (the spine). This spec binds Layers 5–13 together; it adds no new layer — `ExchangeOrder` IS Layer 5 (Proposal/Order), legs live in Layer 8 (Agreement), settlement in 9, fulfillment/outcome in 10–13.
**Companion specs:** [239](./239-intent-spine.md) (intent/match — already object-agnostic), [241](./241-agreement-commitment-registry.md) (agreement — gains the legs structure), [243](./243-payments.md) (monetary-leg adapter), [244](./244-fulfillment.md) (obligation-leg settlement), [272](./272-x402-pay-per-use.md) (a degenerate single-leg exchange; x402 = the HTTP payment adapter), [274](./274-exchange-ontology-and-x402-profile.md) (the formal `apexc:` vocabulary + diagrams + x402 conformance profile for this kernel).

**North star:** *Users express intents, agents form orders, resources settle through adapters, outcomes are proven by receipts, and entitlements become durable access resources. x402 is the HTTP payment adapter for one class of consideration item.*

---

## 0. Why this spec exists

Legacy commerce systems make payment the central object and bolt the deliverable on. Our spine is the opposite — Intent → Order → Agreement → Settlement → Fulfillment → **Outcome/Entitlement** — and the intent layer is already resource-plural (`direction × object`, matcher forbidden from branching on type, spec 239 §7.1). What was missing: a typed consideration structure (what each party owes) and the order artifact that freezes it. Without them every consumer reinvents the shape and money assumptions creep in.

Seaport proves the pattern at scale: an order is arbitrary **offer items** + arbitrary **consideration items with recipients** — never "buyer pays seller". We port that structure, NOT its atomic-token-only assumption: agentic exchange includes off-chain obligations (serve content, run a skill, issue a credential), so our kernel is *token transfer + obligation fulfillment + receipt evidence*, not only token-for-token.

## 1. Decisions

| ID | Decision | Rationale |
|---|---|---|
| **EXC-D1** | **Exchange is the primitive; payment is a leg type.** An agreement body carries `legs: ExchangeLeg[]` (≥1); each leg is a `ResourceItem` with `from`/`to`. The `agreementCommitment` hash covers the full legs array — terms cannot be swapped post-signature. | Names what each party owes; kills the implicit "buyer/seller + price" shape. |
| **EXC-D2** | **Open consideration-type family** (registered, mirrors PMT-2 rails) via `ResourceItem.kind`: `erc20`/`native` (settles via `payments` rail / `PaymentMandate`), `erc721`/`erc1155`/asset-transfer, `service`/`a2a-skill` (settles via spec-244 fulfillment + evidence + validation), `entitlement` (issue/consume), `credential`/`attestation` (spec 242), `content-access`, `compute`, `custom`. New kinds register; the agreement layer never enumerates them. | Resource plurality is open-ended (smart-agent catalog: capital, in-kind, coaching, hospitality, …). |
| **EXC-D3** | **Symmetric settlement, one evidence shape.** Every leg settles through its own adapter but emits the same-shaped evidence: `LegSettled { orderHash, agreementCommitment, legId, kind, evidenceHash, settledAt }` — PaymentReceipt VC for monetary, EvidenceCredential/validation for service, transfer receipt for assets, EntitlementIssued for entitlements. The `OutcomeReceipt` aggregates them: `{ orderHash, intentHash?, status, settled[], fulfilled[], entitlementsIssued[] }`. Outcome closes when all legs settle (or per agreed partial terms). | The car leg and the money leg are audit-equal. Evidence chain = Intent → Order → Mandates → Settlement → Fulfillment → Entitlement → Receipt. |
| **EXC-D4** | **Atomicity is per-exchange policy**, not a money feature: `sequential` (milestone-ordered; X402-D8), `escrowed` (the reserved escrow rail, PMT-6, generalizes to **symmetric escrow** holding any escrowable leg — W2), `atomic-onchain` (all legs on-chain → single redemption, future). On-chain atomicity is optional; **evidence is mandatory**. | Barter needs the same counterparty-risk tools commerce has; escrow must not be money-only. |
| **EXC-D5** | **No money privilege — enforced.** Matcher (239), agreement (241), order, and fulfillment (244) layers MUST NOT branch on a leg's `kind`. Only the leg's adapter knows its settlement details. Pricing/checkout UX is app-layer. | Same load-bearing constraint as 239's "matcher MUST NOT branch on intentType". |
| **EXC-D6** | **`ResourceItem` is the universal unit** (added after review): `{ id, kind, resource (CAIP-19 / agent name / entitlement type / skill id / content URI), amount?, unit?, from?, to?, semantics ('transfer'\|'mint'\|'issue'\|'deliver'\|'execute'\|'consume'\|'attest'\|'grant'\|…), criteria?, adapter, privacy? (public/hash/encrypted fields) }`. Criteria-based items (predicates: "any provider with score > X", "any passage in corpus") reserved for solver lanes. | One type spans tokens, entitlements, credentials, skills, content, compute; the payment rail does not own the exchange model. |
| **EXC-D7** | **`ExchangeOrder` is the Layer-5 artifact; acceptance mints the agreement.** `{ orderId, maker, taker?, offer: ResourceItem[], consideration: ResourceItem[], constraints, zone?, startTime, endTime, salt, nonce, intentHash?, orderHash }`. Maker/taker signatures over `orderHash`; acceptance produces the `agreementCommitment` (241) whose legs = offer ∪ consideration. **Payments never bypass orders:** a payment satisfies a consideration item on an order — `valid payment → serve content` is forbidden; the lane is payment → leg satisfied → zone validates → entitlement issued → content served. `PaymentMandate.contextBinding` gains `orderHash` + `legId` (not merely an HTTP resource). Seaport's `counter`/`salt`/time-window replay model ported; recipient-specific consideration items give licensor/platform/referrer splits for free. | The order is the frozen quote (X402-D9 quote-immutability generalized); AP2's checkout-mandate / payment-mandate separation, resource-general. |
| **EXC-D8** | **Zones = deterministic policy validators; adapters = settlement executors** (Seaport zone/conduit split, generalized). A `ZoneValidator` runs `authorizeBeforeSettlement` / `validateAfterSettlement` against order + intent + receipts (examples: license-scope zone, budget zone, content-privacy zone, registry-trust zone, fee-split zone). A `ResourceAdapter` (`validateItem / prepare / execute / verifyReceipt`) owns HOW a kind settles. On-chain, our caveat-enforcer family (PaymentEnforcer et al.) IS the zone mechanism; a future `IExchangeZone`/`IResourceAdapter` pair arrives only when multi-leg on-chain settlement does. **LLMs propose; zones/adapters/deterministic code verify** (extends 272's "LLMs out of the trust path"). | Policy and execution fail differently and evolve independently; exchange orchestrates adapters, never knows settlement details (EXC-D5). |
| **EXC-D9** | **Entitlements are resources, not auth flags.** `EntitlementSpec { entitlementType, subject, issuer, scope, rights[], constraints (ttl/maxUses/nonTransferable/revocable), provenance { orderHash, paymentReceiptHash } }` — issuable, consumable, revocable, scoped, receipted (issued + consumption receipts), and usable as an offer item, consideration item, or gate input. Private entitlements are VC-shaped (binding `'sa' | 'bearer'` per X402-D7); public anchors are hash-only registry events (`EntitlementIssued`/`EntitlementConsumed`). | The entitlement is the customer-visible product (X402-D8) and the anonymity boundary (X402-D7); making it a resource closes the loop — prepaid credits, transfers, sponsor-paid access are just exchanges over entitlement items. |
| **EXC-D10** | **Product payloads must resolve to the common order** (ERC-7683 principle). Vertical intent/order payloads are allowed, but matchers/solvers consume them only through a resolver producing `ExchangeOrder`. Matching lanes mature in order: V1 quote-acceptance (272's 402 = a one-item quote), V2 standing offers, V3 buyer intents + solver fill, V4 multi-resource bundles, V5 partial fills / credits. | Keeps the marketplace open to verticals without fragmenting the solver surface; partial-fill/credit semantics ride Seaport's proven model. |

**Non-goals:** not a DEX/AMM (no automated price discovery — solver bids do that, Layer 6); not a Seaport clone (no universal atomic-settlement contract in W1); not a legal-contract engine (enforcement is evidence + validation + trust, not courts); not double-entry accounting.

## 2. Reference: smart-agent patterns to port

- **`generalized-intent-matchmaking.md`** — the foundational re-framing: "the system is fundamentally an intent matchmaking platform, of which funding is one specialization among many"; the resource catalog (capital, **in-kind goods**, coaching, mentorship, hospitality, …) maps directly to `ResourceKind` (EXC-D2/D6). **Ported:** resource plurality as the root assumption; money as a specialization.
- **`grants-ontology.md`** — `Pledge` as "subclass of Commitment for capital/in-kind specifically" shows smart-agent already treats money and goods as siblings under one commitment class. **Ported:** consideration as a typed property of the commitment, not a separate payment object.
- **Deliberate divergence:** smart-agent never settles non-monetary legs (engagement/activity tracking only). We add EXC-D3's uniform `LegSettled` evidence so barter legs are as auditable as transfers. Industry patterns (Seaport offer/consideration/zone/conduit/criteria/partial-fill; AP2 mandate separation; ERC-7683 resolver) are design inputs per ADR-0038 — never conformance targets.

## 3. Commerce patterns → legs (the proof of generality)

| Pattern | Legs (ResourceItems) | Settlement |
|---|---|---|
| Traditional ecommerce (goods for money) | `erc20 transfer` (buyer→seller) + `asset-transfer`/`service:shipping` (seller→buyer) | sequential or escrowed |
| Services engagement | `erc20` + `service` (with milestones) | sequential, milestone-bound (X402-D8 pattern) |
| Pure barter (car for service) | `asset-transfer` + `service` | escrowed (asset leg) + milestone (service leg) |
| Donation / grant | single `erc20`/`asset` leg; counter-leg = outcome evidence only | sequential; evidence per smart-agent disbursement pattern |
| x402 pay-per-use (spec 272) | `erc20 transfer` (reader→treasury) + `entitlement issue` + `content-access deliver` (service→reader) | degenerate sequential V1 quote — already conformant via X402-D1/D8 |
| Credential-for-fee (licensing) | `erc20` + `credential issue` | sequential |
| Attribution-for-access, prepaid credits, sponsor-paid | `attestation attest` ↔ `entitlement issue`; `entitlement consume` per read; payer ≠ subject | same kernel, different items (EXC-D9) |

## 4. Requirements (deltas to companion specs / packages)

- **EXC-R1 (`agreements`, spec 241):** agreement body gains `legs: ExchangeLeg[]` (= ResourceItems with from/to); `agreementCommitment` covers it; JV stores the full legs; on-chain stays commitment-hash-only (D-46.3 unchanged).
- **EXC-R2 (`payments`, spec 243):** `PaymentMandate.contextBinding` gains optional `orderHash` + `legId` binding the mandate to the consideration item it settles (EXC-D7). The monetary-leg adapter IS the existing rail registry — no other changes; x402 stays the HTTP payment adapter.
- **EXC-R3 (`fulfillment`, spec 244):** a FulfillmentCase tracks all legs; `LegSettled` + `OutcomeReceipt` (EXC-D3) are typed evidence kinds; outcome predicates ("entitlement-issued", "content-delivered", "resource-transferred") close the case — the audit question is *did the promised outcome become true*, not *did a payment happen*.
- **EXC-R4 (registry kit / ADR-0038):** `ResourceKind`s and adapters are registered families like payment rails; verticals add kinds (e.g. `in-kind:equipment`) without spec change.
- **EXC-R5 (spec 272):** no wire/contract change — the lbsb 402 quote is a V1 single-consideration-item order; `quoteId` is the degenerate `orderHash`. When the kernel lands, 272's gate becomes the lbsb license **zone** and the entitlement mint becomes an `entitlement issue` offer item.
- **EXC-R6 (packages, when implementation starts):** kernel types (`ResourceItem`, `ExchangeOrder`, zones, adapter interfaces, receipts) land as `@agenticprimitives/exchange` (dependency-light, importable by `payments`/`fulfillment`/`intent-marketplace`); entitlement spec/issue/consume/verify graduates to `@agenticprimitives/entitlements` when a second consumer exists. Both Ring-0-pure (ADR-0037). Spec-before-code applies; this spec is that spec for the types, but package scaffolding awaits a wave plan.
- **EXC-R7 (contracts, future wave):** no universal exchange contract in W1. When multi-leg on-chain settlement is needed: a minimal `AgenticExchange` kernel (signature/time/counter checks, zone hook, adapter dispatch, `ExchangeSettled`/`OutcomeFulfilled` hash-only events) + `EntitlementRegistry` (hash-only `EntitlementIssued`/`EntitlementConsumed`). Until then the delegation stack (272 W1 contracts) is the settlement path.

## 5. Invariants (DO NOT BREAK)

- **EXC-INV-1** — every agreement has ≥1 leg; every leg has a typed ResourceItem, parties, and an adapter. Un-typed consideration is a substrate violation.
- **EXC-INV-2** — the order/commitment hash covers all legs; no leg added, dropped, or re-priced post-signature (quote-immutability generalized from X402-D9).
- **EXC-INV-3** — no spine layer outside adapters branches on `kind` (EXC-D5).
- **EXC-INV-4** — every settled leg emits `LegSettled` evidence regardless of kind; monetary legs get no extra audit standing, non-monetary legs no less.
- **EXC-INV-5** — atomicity policy is declared in the order (covered by the hash); adapters/zones enforce it; UI never invents it.
- **EXC-INV-6** — payments never bypass orders: a settlement that doesn't satisfy a consideration item on a hashed order is invalid (EXC-D7). Degenerate single-item quotes (272) count as orders.
- **EXC-INV-7** — entitlements are issued/consumed/revoked only through receipted operations with provenance (`orderHash`); never a bare boolean in app logic (EXC-D9).
- **EXC-INV-8** — orders, receipts, and on-chain events carry hashes/scopes only — no licensed content, user queries, or sensitive URLs (extends PAY-WIRE-6).
- **EXC-INV-9** — LLMs/agents may discover, negotiate, and propose orders; only deterministic zones/adapters validate intent, order, payment, entitlement, and fulfillment (EXC-D8).

## 6. Implementation status

Doctrine + type shapes only (this spec). Order of arrival: EXC-R1 with the next `agreements` wave; EXC-R2's `orderHash`/`legId` fields whenever 243 types are next touched (additive); EXC-R3 with `fulfillment`; the `exchange` package (EXC-R6) and symmetric escrow (EXC-D4) per a future wave plan; contracts (EXC-R7) only when multi-leg on-chain settlement is demanded. Spec 272 ships unchanged — it is the V1 quote-acceptance lane of this kernel. Follow-on specs (entitlements-as-resources detail, exchange settlement contracts) are reserved and should be cut from this spec only when their implementation waves start.
