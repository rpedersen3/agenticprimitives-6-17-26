# Spec 220 — Agent identity bootstrap process

**Status:** v0 (2026-05-24).
**Owner:** cross-package — touches `agent-account`, `agent-naming`,
`agent-profile`, `connect-auth`, `custody`, and downstream demos.
**Architecture commitment:**
[ADR-0010 — Smart Agent address is the canonical identifier](../docs/architecture/decisions/0010-smart-agent-canonical-identifier.md).
**Related ADRs:** 0006 (naming as resolution), 0007 (identity stack
shape), 0008 (CAIP-10 nativeId), 0009 (on-chain ontology / SHACL).

---

## 1. Purpose

Define the sequenced process for creating, naming, custodying, and
facet-registering a new Smart Agent so every actor in the system (a
person, an org, a service agent, a treasury) lands with a coherent,
forced-unique identity that other packages and applications can rely
on. The Smart Agent address is the canonical identifier; everything
else is a facet registration that points at it.

## 2. Scope

In scope:

- The bootstrap sequence (deploy → name → custody → facets).
- Forced-unique name discovery rules (number-suffix counter).
- Authentication resolution rules (credential → canonical SA).
- Recovery / rotation invariants (the canonical SA address never
  changes).
- Demo / SDK implementation requirements.

Out of scope (covered in their own specs):

- Custody policy mechanics → spec 207 / 209 / 213.
- Naming registry contract internals → spec 215.
- Identity profile schema → spec 217.
- Relationship-edge mechanics → spec 216.

## 3. The canonical identifier

The **ERC-4337 Smart Agent (SA) address** is the canonical identifier
for any actor. It is:

- **Deterministic.** CREATE2 from the factory + salt + init code.
- **Stable across rotation.** Passkey/EOA custodian changes never
  change the SA address.
- **Self-describing on chain.** The SA's deployed bytecode + custody
  policy answer "who can act for this identity".
- **Universal across our stack.** Every package keys off it.

Other identity layers are **facets** that point AT the canonical SA:

| Facet | Identifier | Backlink to canonical SA |
| --- | --- | --- |
| `.agent` name (agent-naming) | namehash node | `addr` record + `setPrimaryName` |
| AgentCard (agent-profile) | `metadata-hash` + URI | `metadata-uri` record + on-chain anchor at the SA's subject |
| ERC-8004 trustless agent | registry-issued id | `agent_address` field |
| GoDaddy ANS handle | handle string | resolver `addr` |
| HCS-10 channel / HCS-11 profile | Hedera topic id | `accountId` field (CAIP-10) |
| HCS-14 UAID | UAID string | derived from canonical SA's CAIP-10; not generated by us (ADR-0008) |
| DID document | DID URI | `verificationMethod[].controller` |

## 4. The bootstrap sequence

Every new agent (person, org, service, treasury) goes through these
six steps in order. Steps 1, 2, 3, 4 are MANDATORY for a coherent
identity. Steps 5, 6 are optional and may be deferred.

```
1. Deploy Smart Agent              → canonical address acquired
2. Register name (forced-unique)   → naming facet
3. Set primary name                → reverse record (round-trip)
4. Enroll custodians                → passkey + SIWE EOA → SA authority
5. Publish profile                  → AgentCard facet (optional)
6. Register external facets         → ERC-8004 / ANS / HCS (optional)
```

### Step 1 — Deploy Smart Agent

- `agent-account` package (`AgentAccountFactory.createAgentAccount`).
- CREATE2 salt derived from the chosen auth methods + a stable user
  scope. **Salt must NOT include the chosen name** — the name is a
  facet, not an identity input (ADR-0006 / ADR-0010).
- Resulting address is the canonical identifier from this point on.

### Step 2 — Register name (forced-unique)

- `agent-naming` package + the `PermissionlessSubregistry` contract.
- The chosen base label SHOULD be human-readable and tied to the
  user's intent (`alice`, `acme`, `treasury`).
- If the base label is already taken in the parent namespace, the
  helper MUST append a sequential counter to find a free label:
  `alice` → `alice2` → `alice3` → … (stop at a sensible cap like 100
  attempts).
- The forced-unique label MUST be deterministically derivable from the
  base label + the registry state — NOT salted with a random or
  address-derived suffix that loses the human-readable narrative.
- The registered name's `addr` record points at the canonical SA
  address.

### Step 3 — Set primary name

- `AgentNameRegistry.setPrimaryName(node)`.
- Called BY the canonical SA (msg.sender == SA address).
- Round-trip discipline: after this step,
  `universalResolver.reverseResolve(canonicalAddr) == node`. If forward
  and reverse records disagree, reverseResolve returns null and the
  name does not display.

### Step 4 — Enroll custodians

- `agent-account` (passkey via `addPasskey` / EOA via `addCustodian`).
- Each enrolled passkey credential + SIWE EOA can authorize operations
  for the canonical SA via the SA's CustodyPolicy quorum.
- The SAME credential MAY be enrolled on multiple SAs; the
  authentication layer disambiguates by picking the active SA.

### Step 5 — Publish profile (optional)

- `agent-profile` package + `AgentProfileResolver` contract.
- Subject = `bytes32(uint256(uint160(canonicalAddr)))`.
- The profile JSON's canonical-JSON content hash is anchored on chain
  at the SA's subject under the `atl:metadataHash` predicate, with
  the URI under `atl:metadataURI`.

### Step 6 — Register external facets (optional)

- ERC-8004 trustless-agents registry: register the SA with whatever
  the registry's required fields are; the `agent_address` MUST equal
  the canonical SA address.
- GoDaddy ANS: register a handle; the `addr` record MUST point at
  the canonical SA address (via CAIP-10 if cross-chain).
- HCS-10 / HCS-11: create the appropriate Hedera topics; the
  `accountId` field references the canonical SA via CAIP-10.
- DID documents: the `verificationMethod[].controller` MUST be the
  canonical SA address.

In every case, the back-link is mandatory. A facet that doesn't
reference the canonical SA is not a legitimate facet — it's a
parallel-authority risk and the demo / SDK MUST refuse to write it.

## 5. Forced-unique name discovery

The reference algorithm:

```ts
async function findUniqueLabel(baseLabel: string, parentNode: Hex): Promise<string> {
  for (let i = 1; i < 100; i++) {
    const candidate = i === 1 ? baseLabel : `${baseLabel}${i}`;
    const labelhash = keccak256(toHex(candidate));
    const childNode = await registry.childNode(parentNode, labelhash);
    if (childNode === ZERO_NODE) return candidate;
    const recordExists = await registry.recordExists(childNode);
    if (!recordExists) return candidate;
  }
  throw new Error(`Could not find unique label after 99 attempts starting from ${baseLabel}`);
}
```

Implementation notes:

- The `childNode == 0` short-circuit is the fast path for the first
  user to claim a label.
- The `recordExists` check covers the edge case where childNode is
  non-zero but the record was somehow not registered (defensive).
- The cap (100 attempts) protects against runaway loops; in practice
  the loop ends after 1-2 iterations.

## 6. Authentication resolution

Given a passkey credential or SIWE EOA, the auth flow:

1. Compute the passkey's `pia` (passkey identity address, a CREATE2
   from the P-256 pubkey) OR take the SIWE EOA directly.
2. Look up which Smart Agents have that identity as a custodian.
   - On-chain: walk `CustodianAdded` / `PasskeyAdded` events filtered
     by the identity.
   - Off-chain index (recommended for production): a worker maintains
     a `custodian → SA[]` reverse map.
3. If exactly one SA matches → it is the canonical SA.
4. If multiple SAs match → the auth layer prompts the user to pick.
5. Subsequent session is keyed by the canonical SA address. The
   passkey / EOA appears as the signer principal but the authority is
   always the SA's.

## 7. Recovery / rotation invariants

When a custodian rotates (passkey lost + replaced, EOA disconnected,
custody policy quorum changes), the canonical SA address MUST remain
the same. Therefore:

- `reverseResolve(canonicalAddr)` continues to resolve to the same
  name.
- The published profile (metadata-uri + metadata-hash) continues to
  anchor at the same SA subject.
- Relationship edges where the SA is the subject or object continue
  to reference the same address.
- External facet registrations (ERC-8004, ANS, HCS) continue to
  point at the same SA.

Recovery is therefore an entirely internal CustodyPolicy operation;
no facet has to know about it.

## 8. Implementation requirements

### Packages

- **agent-account**: `AgentAccountFactory.createAgentAccount` MUST
  compute the salt from auth methods + scope only — NEVER from a
  name (ADR-0010 § 2).
- **agent-naming**: the SDK MUST expose a `findUniqueLabel` helper
  that implements the forced-unique algorithm in § 5. The
  `claimPsaName`-style high-level helper MUST take a base label and
  return the registered (possibly suffixed) full name.
- **agent-profile**: profile-publish helpers MUST refuse to set
  `metadata-uri` without a corresponding `metadata-hash` (anti-mutation
  invariant). The SA address is the subject — never the name.
- **connect-auth**: session JWTs MUST include the canonical SA
  address as the primary subject; the passkey credentialId / EOA
  appears as a signer principal claim only.
- **custody**: unchanged — authority lives here; this spec only
  requires that authority always traces to the SA.

### Demos

- `demo-web-pro` and `demo-web-recovery` MUST follow the full
  six-step sequence for every actor (person seats, Org, Treasury).
- The auto-claim helper MUST use number-suffix uniqueness (not a hex
  salt) so the name reads naturally (`alice`, `alice2`, …).
- The success card UI MUST surface the registered name immediately
  after step 3 completes.
- NameDisplay components MUST resolve EVERY Smart Agent address
  through reverseResolve, falling back to truncated hex when no
  primary name is set.

## 9. Operational guidance

- **Don't burn human-readable labels on failed runs.** The
  number-suffix algorithm makes `alice` available to the first
  successful claimer regardless of how many attempts came before. We
  do NOT pre-allocate labels for users who haven't completed step 3.
- **Don't make label uniqueness state-dependent on demo session.**
  Two different browsers should both be able to register some `alice*`
  label; the algorithm naturally serializes them into `alice`,
  `alice2`, `alice3`, etc.
- **Don't conflate facet identity with canonical identity in audit
  logs.** Audit events MUST include the SA address; the name is
  decorative.
- **Don't introduce a UAID generator.** Per ADR-0008, consumers derive
  UAIDs locally; we expose only the CAIP-10 `nativeId`.

## 10. Phase plan

| Phase | Scope | Status |
| --- | --- | --- |
| **P1** | ADR-0010 + this spec land. Demos refactored to use the number-suffix algorithm. CLAUDE.md docs reference the principle. | **shipped 2026-05-24** |
| **P2** | connect-auth: emit canonical SA address in JWT claims; demo apps consume that. SDK helper `findUniqueLabel` exposed from agent-naming. | next |
| **P3** | Off-chain custodian-to-SA reverse index (worker route + cache). Used by connect-auth Phase 2 + demo a2a. | after P2 |
| **P4** | Facet integrations — ERC-8004, HCS-10/11, ANS — each documented separately + wired into demos that need them. | per-facet, on demand |

## 11. Out of scope

- A global on-chain custodian-to-SA reverse index — out of scope for
  now (off-chain event-indexing is sufficient).
- Multi-chain canonical identity (one SA, many chains) — deferred to
  a future spec; CAIP-10 nativeId is the bridge for now.
- ANS-style "primary handle" semantics where the SA picks ONE facet
  as canonical-display — that's a UI choice, not a doctrine.
