R5.10 — Session JWT `iss` / `aud` / `sid` binding (P1-1 closure).

### Breaking

- **`JwtClaims` extended with required fields:** `iss`, `aud`,
  `sid`, plus optional `nonce`. Pre-R5.10 sessions cannot be minted
  without supplying these (sid is auto-generated when omitted, but
  the type signature requires explicit `iss` + `aud`).
- **`verifySession` signature changed.** Now takes an optional
  second arg `opts?: VerifySessionOpts` with `expectedIss`,
  `expectedAud`, `clockSkewSec`, `developmentMode`. **In
  `NODE_ENV=production` (and without `developmentMode: true`),
  `verifySession` THROWS if `expectedIss` or `expectedAud` is
  missing.** Tests / dev paths opt out via `developmentMode: true`.

### Why

External senior-architect audit P1-1: pre-R5.10 session cookies
carried no `iss` (issuer) or `aud` (audience), so multiple tokens
issued for different relying apps with the same HS256 key were
mutually replayable. There was also no future-iat skew check, so a
misconfigured / malicious mint server could issue tokens with iat
well in the future, outliving the TTL window.

### What changed

- Added `iss` (issuer URI) and `aud` (audience id or id[]) to claims;
  verifier cross-checks both when `expectedIss` / `expectedAud` opts
  are supplied.
- Added `sid` (high-entropy session id) for per-session revocation;
  auto-generated by `mintSession` when not supplied.
- Future-iat rejection: `claims.iat - clockSkewSec > now` → null.
- Default `clockSkewSec = 30` (OIDC reference).
- Production gate: `verifySession` throws when `expectedIss` /
  `expectedAud` are missing in `NODE_ENV=production`.

### Tests

- 21 new R5.10 tests in `test/unit/sessions.test.ts` covering iss
  match + reject, aud-as-string vs aud-as-array, sid auto-gen +
  caller-supplied + uniqueness, production gate (throws on missing,
  developmentMode bypass), future-iat reject beyond skew, accept
  within skew, accept-expired-within-skew, reject-expired-beyond-skew.
- 102/102 connect-auth tests pass (was 81; +21 R5.10).

### Migration

Consumers must add `iss` and `aud` to every `mintSession` call:

```ts
mintSession({
  // ...existing claims...
  iss: 'https://broker.example.com',
  aud: 'https://relying-app.example.com',
});
```

And pass them to `verifySession` for production reads:

```ts
verifySession(cookie, {
  expectedIss: 'https://broker.example.com',
  expectedAud: 'https://relying-app.example.com',
});
```

Tests can keep working unchanged by omitting opts (verifier defaults
to signature + exp + future-iat check only, without iss/aud
binding) — but should add `developmentMode: true` if they run with
`NODE_ENV=production` set.

### demo-a2a

Two callsites updated in `apps/demo-a2a/src/index.ts`:
- `smartAccountFromCookie` uses `verifySession(cookie, { developmentMode: true })`
  with a clear comment that spec 227 (Real-Connect) will replace the
  demo-mode bypass with real iss/aud bindings.
- `/auth/siwe-verify` populates `iss` + `aud` from the new
  `CONNECT_BROKER_ORIGIN` env var (default `https://demo-a2a.local`).
